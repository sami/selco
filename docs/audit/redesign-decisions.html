0. TMA 01 Output — Framed as a Prototype

The first iteration of TMA 470 delivered a working prototype rather than a production application. This is appropriate for the planning and discovery phase, and consistent with the iterative lifecycle chosen in Section 4 of TMA 01. The table below records what the prototype validated, invalidated, and left open — providing a traceable basis for the TMA 02 design decisions that follow.

Prototype Outcomes (Weeks 1–4, Feb–Mar 2026)

✅ Validated

  • Astro + React islands delivers fast static HTML with selective hydration
  • Three-layer separation (pure logic → configs → pages) works; same functions reused in standalone pages and wizard without modification
  • Stateless application model is sufficient for trade use (per-job calculation, no need to persist)
  • Mobile-first layout: TSA survey confirmed on-site phone as primary device
  • Multi-material wizard concept: colleague interest on internal platform confirmed demand
  • 100% logic-layer test coverage is achievable and catches real bugs (IEEE 754 floating-point)
  • SELCO branding feasible using publicly available brand colours

❌ Invalidated

  • "Save to project" state persistence — added complexity without user value; removed in Week 3 product pivot
  • Vite + React SPA — pivoted to Astro in Week 2 for static performance and hydration control
  • Single-file calculator approach — organic growth to three-layer architecture revealed the need for explicit separation of concerns from the outset
  • Three competing CSS token systems are unworkable — phantom tokens cause invisible interactive elements

⚠️ Open / TMA 02 to address

  • Post-implementation TSA feedback — evaluation not yet run
  • Masonry, concrete, paint, insulation calculators not yet built
  • Product range mapping (bag sizes → purchase quantities) partially implemented
  • Accessibility audit findings not yet fixed (WCAG AA target not yet met)
  • Focus ring, button visibility, contrast issues identified but not resolved

Evidence base: Prototype audit (docs/audit/audit-report.html), UX evaluation (docs/audit/ux-evaluation.html), TMA 01 progress log (Appendix 2, Weeks 1–4), Ju's PT3 feedback (57/100, March 2026).

1. Architecture Decisions

ID Area Current State Problem Redesign Decision Rationale
A-1 P0 CSS token system Three competing systems: (1) Tailwind @theme in global.css with real tokens, (2) SelcoLayout.astro inline :root with 11 hardcoded hex variables, (3) 12 phantom tokens used in CalculatorLayout.tsx and wizard that are never defined anywhere Phantom tokens (--color-brand-blue, --color-brand-yellow, --color-surface, --color-border, --radius-card, etc.) resolve to transparent / no value. This causes: invisible Next button, invisible Calculate button, invisible focus rings, invisible results text, invisible error alerts. Consolidate to a single @theme block in global.css. Define all 12 currently-phantom tokens. Deprecate the SelcoLayout inline :root block and migrate its values into @theme. All components use only Tailwind utility classes referencing @theme tokens. Tailwind v4's @theme is the canonical design token mechanism. A single source of truth prevents the token drift that caused B-2, F-6, and 5 other visual failures. Aligns with TM354's separation of concerns principle.
A-2 P1 Layout files Two layout files: BaseLayout.astro (Tailwind-based, used by early pages) and SelcoLayout.astro (inline CSS, used by all current pages). BaseLayout is effectively unused. Dual layout files create divergent styling contexts. Components styled for one layout may not work in the other. Increases maintenance surface. Extend SelcoLayout.astro to source all styling from the unified @theme tokens (resolved by A-1). Remove BaseLayout.astro or archive it. All pages use SelcoLayout. The SELCO chrome (header, nav strip, info bar, footer) in SelcoLayout is correct. The problem is the styling approach, not the component structure. Removing the duplicate reduces the risk of future token drift.
A-3 P1 CalculatorLayout component CalculatorLayout.tsx is a well-designed shared React component: two-column form/results layout, field groups, error handling, empty state. Used by Coverage Calculator and Unit Converter. Not used by the Tiling Wizard. All styling in CalculatorLayout.tsx references phantom tokens (see A-1). The component is architecturally correct but visually broken. Keep the component. Replace all phantom token references with the unified tokens defined in A-1. Consider extending it to handle the tiling wizard layout so there is one shared calculator shell. The component is a good abstraction: it enforces consistent form/results structure across calculators. The fix is purely the token references, not the architecture. Reuse aligns with TM352's DRY principle.
A-4 P3 Three-layer separation Pure calculation logic (TypeScript functions) → project configuration objects → Astro page routes. The same pure functions are used in both standalone calculator pages and wizard steps without modification. 46 passing tests at 100% logic coverage. No problem. This architecture is working correctly and is validated by the prototype. Keep unchanged. Continue applying the three-layer pattern to all new calculator modules (masonry, concrete, paint, insulation). The tiling wizard reused existing standalone calculator logic without modification — this is the pattern working as intended. Aligns with TM352 component-based design and TM354 separation of concerns.
A-5 P2 Navigation architecture SelcoNav (SelcoNav.astro): product category bar with 11 items, all pointer-events: none / cursor: not-allowed. Purely decorative. SelcoSidebar: actual working navigation — Home, Unit Converter, Tiling (live), Masonry (disabled), Concrete (disabled). Coverage Calculator is absent from the sidebar entirely. Two navigation systems with completely different behaviours. Users expect the top nav to be interactive. Coverage Calculator is unreachable from any navigation — only via the homepage grid. Disabled sidebar items (Masonry, Concrete) have no "Coming soon" label and look broken. SelcoNav: Add a working "Calculators" item that links to the homepage or opens a dropdown listing all live tools. Keep the category items decorative (mirrors real Selco site) but make one item functional.

SelcoSidebar: (1) Add Coverage Calculator to the sidebar. (2) Add a "Coming soon" badge to disabled items so they visually communicate intent rather than appearing broken.		 N-1, N-3 findings. Tidwell (2020) Clear Entry Points: every major feature must be discoverable from the primary navigation. The Coverage Calculator existing without being navigable is a discoverability failure.
A-6 P2 Breadcrumb semantics Breadcrumbs rendered as a <div> with inline <a> and <span> elements in SelcoLayout.astro. Non-semantic structure. Screen readers cannot identify the breadcrumb as a landmark. Fails WCAG 2.4.8 (Location) best practice. Replace with <nav aria-label="Breadcrumb"><ol><li> structure. Add aria-current="page" to the final crumb. N-4 finding. Semantic HTML is zero-cost in effort and directly improves accessibility score (currently 83/100 on Lighthouse).

2. UX Decisions

ID Area Current State Problem Redesign Decision Rationale
U-1 P1 Wizard progress indicator The wizard has a segmented progress bar (a row of coloured/uncoloured div strips) and a text label "Step X of Y — [Name]". The segments are not clickable. Navigation is Next/Back only. Progress segments use bg-brand-blue (phantom token — invisible). N-2 finding. Users cannot see named steps or jump to a previous step. Progress bar is invisible because the token is undefined. Tidwell's Sequence Map pattern is not implemented — only navigation arrows. Replace the strip bar with a named step indicator: each step shown as a pill or numbered label with its name. Completed steps are clickable (navigate back). Current step is highlighted. Future steps are muted. Fix bg-brand-blue to resolved token (A-1). N-2, B-2 findings. Tidwell (2020) Sequence Map: "Show users where they are in a multi-step process and let them navigate back to completed steps." Nielsen's visibility of system status heuristic. Directly addresses Ju's "vague idea of how your app will work" — a visible wizard structure communicates the tool's scope.
U-2 P1 Wizard step validation The "Next" button on Step 1 (Setup) advances even when width/height inputs are empty. Subsequent steps calculate using 0 as input, producing incorrect results silently. No required field validation on any wizard step. F-7 finding. A user who accidentally skips Step 1 inputs receives a summary with all quantities showing 0. There is no error message and no indication that the calculation is invalid. Add step-level validation on the Next button: required fields must have valid positive values before advancing. Show an inline error message under the invalid field. Do not use window.alert(). F-4, F-7 findings. Tidwell (2020) Same-Page Error Messages: errors should appear adjacent to the field that caused them. Forgiving Format: accept decimal inputs with locale flexibility. Krug (2014): errors must be obvious and immediate.
U-3 P1 Escape hatch from calculators No "← Back to calculators" link on any calculator page. The breadcrumbs infrastructure exists in SelcoLayout but breadcrumbs are only rendered when the page passes the breadcrumbs prop. N-5 finding. Users arriving deep in the wizard have no navigational escape other than the browser back button. Tidwell's Escape Hatch pattern is missing. Ensure all calculator pages pass populated breadcrumbs to SelcoLayout: at minimum [Home → Calculators → Current page]. The breadcrumb will serve as the escape hatch. Fix breadcrumb semantics per A-6. N-5 finding. Tidwell (2020) Escape Hatch: "Always give users a way out." The infrastructure already exists — this is a configuration fix, not a new feature.
U-4 P2 Mobile menu accessibility The mobile menu drawer (in SelcoHeader.astro) opens/closes via a hamburger button. There is no focus trap when the drawer is open, no Escape key handler, and no aria-modal attribute. N-6, A-5 findings. Keyboard users can Tab through the entire page while the overlay is open. Screen readers do not announce the drawer as a modal context. Fails WCAG 2.1.2 (No Keyboard Trap) in the inverse direction — there is no trap when there should be one. Add a focus trap to the open drawer: Tab/Shift+Tab cycle through focusable elements within the drawer only. Add an Escape key handler that closes the drawer and returns focus to the hamburger button. Add aria-modal="true" to the drawer element. N-6, A-5 findings. WCAG 2.1 criterion 2.1.2. Standard pattern for any modal overlay. Relatively low implementation effort with significant accessibility impact.
U-5 P2 Form error handling Error handling is inconsistent across calculators: CalculatorLayout uses an ErrorAlert component (but without role="alert"). One calculator uses window.alert(). ProjectListsCalculator uses placeholder-only inputs with no visible labels. Errors appear only on form submission, not on blur. F-2, F-5 findings. window.alert() is inaccessible and breaks mobile UX. Missing role="alert" means screen readers do not announce errors. Placeholder-only fields fail WCAG 1.3.5. (1) Add role="alert" and aria-live="polite" to all error display components. (2) Replace window.alert() with inline error messages. (3) Add visible <label> elements to all inputs in ProjectListsCalculator. (4) Add aria-describedby linking each input to its error message element. F-2, F-5, A-4 findings. WCAG 1.3.1 (Info and Relationships), 3.3.1 (Error Identification). Tidwell (2020) Same-Page Error Messages.
U-6 P2 Skip link SelcoLayout.astro has no skip navigation link. BaseLayout.astro (now unused) had one. A-2 finding. Keyboard and screen reader users must Tab through the entire header, nav strip, and info bar before reaching the main content on every page load. Add a visually hidden skip link as the first focusable element in SelcoLayout: <a href="#main-content" class="sr-only focus:not-sr-only">Skip to main content</a>. Add id="main-content" to the <main> element. A-2 finding. WCAG 2.4.1 (Bypass Blocks). Standard two-line fix. Was present in the original BaseLayout and was lost in the SelcoLayout migration.
U-7 P3 Touch target sizes Some toggle buttons (unit switcher m²/ft² pills in the wizard, some icon buttons) render at below 44×44px. A-7 finding. WCAG 2.5.5 (Target Size) recommends minimum 44×44px for touch targets. Small targets increase error rate on mobile, which is the primary use context. Enforce min-h-[44px] min-w-[44px] on all interactive buttons. Specifically fix the unit toggle pills in the wizard and any icon-only buttons. A-7 finding. On-site mobile use confirmed as primary context (TSA survey). Wroblewski (2011) Mobile First: touch targets must be appropriately sized for finger input.

3. Visual Design Decisions

Overall design approach: Minor tweaks, not a fresh template

The SELCO branding is already in place and correctly reflects the real selcobw.com visual identity. The redesign does not require a visual overhaul. All five decisions below are fixes to inconsistency and token gaps — the design direction (dark blue, yellow accent, Open Sans, sidebar layout) is correct and should be kept.
ID Area Current State Problem Redesign Decision Rationale
V-1 P0 Brand colour tokens Four different shades of blue used inconsistently: #003087 (global.css @theme --color-primary), #004B8D (SelcoLayout --selco-blue), #1b3a6e (SelcoLayout --selco-nav-blue), #00264d (dark blue). Yellow: #FFCD00 in SelcoLayout but never defined in @theme. ST-3 finding. Multiple blue variants produce visual inconsistency. Yellow not in @theme means it is unavailable as a Tailwind utility class, forcing hardcoded hex values throughout. Define a canonical colour palette in @theme:
--color-brand-primary: #003087 (Selco Blue)
--color-brand-nav: #1b3a6e (nav bar — structurally distinct)
--color-brand-dark: #00264d (dark blue)
--color-brand-accent: #FFCD00 (Selco Yellow)
Remove --selco-blue, --selco-nav-blue, --selco-dark-blue, --selco-yellow from SelcoLayout inline block. Replace all usages with Tailwind utilities. 		ST-3, A-1 findings. A single authoritative colour palette is a prerequisite for consistent design and working interactive states. Resolves the token drift root cause identified in A-1.
V-2 P0 Button system 7+ distinct button variants: bg-brand-blue (phantom), bg-brand-yellow (phantom), btn-primary (global.css), btn-secondary, hard-coded blue hex in SelcoLayout footer, ghost text-links styled as buttons, disabled variants. No consistent design system. B-1 finding. Visual inconsistency across calculators. Two primary action buttons (Next, Calculate) are invisible because they use phantom tokens. No predictable interaction model. Define 3 canonical button variants as @layer components utilities in global.css:
.btn-primary: brand-primary background, white text (navigational actions)
.btn-accent: brand-accent (yellow) background, brand-dark text (primary CTAs: Calculate, Next, Start Over)
.btn-ghost: transparent, brand-primary text (secondary actions: Reset, Skip)
Delete all phantom-token button classes. Apply the three variants consistently. 		B-1, B-2 findings. Three variants cover all use cases in the current application. Fewer variants reduce cognitive load (Krug, 2014) and maintenance cost. The accent button (yellow on dark blue) matches SELCO's visual identity for CTAs.
V-3 P1 Typography system Three competing font families: system-ui declared in global.css @theme, 'Open Sans', Helvetica, Arial declared in SelcoLayout body (but Open Sans is never imported), 'Arial Black' for page titles in SelcoLayout. ST-1 finding. Open Sans is declared but not loaded — all SelcoLayout pages fall back to Helvetica, creating a font mismatch between the declared intent and what renders. The system-ui in global.css and the unloaded Open Sans are different fonts. Add a Google Fonts import for Open Sans (400, 600, 700, 800) to SelcoLayout.astro <head> with display=swap. Update global.css @theme --font-sans to include 'Open Sans' as first preference. Keep 'Arial Black' for .page-title display headings only. ST-1 finding. Open Sans is the declared intent and the correct choice for a professional trade tool. Loading it removes the font fallback inconsistency. display=swap avoids render blocking (addresses the Lighthouse performance opportunity for font loading).
V-4 P1 Colour contrast Yellow #FFCD00 on blue #004B8D background produces a contrast ratio of approximately 2.5:1. Used on the info strip text and in button label contexts. A-1 finding. WCAG 2.1 Level AA requires 4.5:1 for normal text (≤18pt / non-bold), 3:1 for large text (≥18pt or bold ≥14pt). 2.5:1 fails both thresholds for normal-weight text. For small text on coloured backgrounds: use white on brand-primary (#ffffff on #003087 = 8.6:1 ✅) instead of yellow on blue. Reserve yellow-on-blue only for large/bold display text where 3:1 suffices. For the accent button (.btn-accent): use brand-dark (#00264d) text on yellow background (#FFCD00 on #00264d = ~6.5:1 ✅). A-1 finding. WCAG 2.1 Success Criterion 1.4.3. The fix also makes button text more legible for users in bright outdoor environments (on-site mobile use).
V-5 P2 Focus ring system Two focus ring definitions exist: @utility focus-ring in global.css (working — uses --color-primary/50, a real token) and focus:ring-2 focus:ring-[--color-brand-blue]/30 in CalculatorLayout.tsx (broken — --color-brand-blue is phantom). F-6 finding. Focus is invisible on all inputs in calculators using CalculatorLayout. Fails WCAG 2.4.7 (Focus Visible). Delete the phantom focus ring in CalculatorLayout.tsx. Apply the global focus-ring utility class consistently to all interactive elements (inputs, selects, buttons). After V-1 defines --color-brand-primary, update focus-ring to use it as the ring colour. F-6, A-6 findings. WCAG 2.4.7. A single consistent focus ring is simpler to maintain and produces a uniform keyboard navigation experience. The working focus-ring utility already exists — this is a removal and application, not a new implementation.

4. Technology Decisions

ID Area Current State Problem / Opportunity Decision Rationale
T-1 P3 Tailwind CSS v4 Already using Tailwind v4 with @theme, @layer, and @utility. A build issue with the focus-ring utility was resolved in Week 4. No problem. The v4 approach is correct and the build issue is resolved. Keep unchanged. Continue using Tailwind v4 @theme as the token system. Do not downgrade to v3. Tailwind v4's @theme is more powerful than v3's theme.extend — it generates CSS custom properties that can be referenced from any CSS context, not just Tailwind utilities. This is the mechanism needed to fix the phantom token problem.
T-2 P0 Phantom token audit 12 undefined CSS custom properties used across 8+ component files. Identified: --color-brand-blue, --color-brand-yellow, --color-surface, --color-surface-foreground, --color-border, --color-muted-foreground, --color-success, --color-destructive, --radius-card, --radius-input, --radius-button. These tokens produce silent failures — no build error, no console warning. The application appears to build successfully but renders with invisible interactive elements. Add a CI lint step that greps for undefined CSS custom property references. Run as part of the GitHub Actions build. Additionally: resolve all 12 phantom tokens in global.css @theme as the immediate fix. T-2 finding from code audit. A lint step prevents this category of silent failure from recurring in future iterations. Without it, adding new component styles could reintroduce phantom tokens. Low implementation effort (a grep script in the CI workflow).
T-3 P2 Accessibility testing No automated accessibility testing in the test suite or CI pipeline. Accessibility issues (A-1 through A-7) were found manually during the UX audit. Manual accessibility testing found 7 distinct issues. Without automated testing, these can silently regress between iterations. Add axe-core as a Playwright integration (or standalone node script) in the CI pipeline. Run against the deployed GitHub Pages URL on each push. Fail the build on any new critical or serious violations. A-1 through A-7 findings. The target is WCAG 2.1 Level AA (stated in TMA 01 scope). Automated testing creates a regression safety net. axe-core is the industry-standard tool used by the BBC, GOV.UK, and NHS Digital. Low false-positive rate for critical/serious violations.
T-4 P2 Font loading Open Sans declared in SelcoLayout body font stack but not imported. Font Awesome loaded from CDN as a render-blocking stylesheet (cdnjs.cloudflare.com/...all.min.css, the entire 1000+ icon bundle). Open Sans not loading = Helvetica fallback on all SelcoLayout pages (ST-1). Font Awesome full bundle = ~150KB of unused CSS, contributing ~2200ms to FCP (Lighthouse performance opportunity). (1) Add Google Fonts import for Open Sans in SelcoLayout.astro (see V-3). (2) Defer Font Awesome: either switch to Lucide React (tree-shakeable, zero unused CSS) or add media="print" onload="this.media='all'" to the Font Awesome link to prevent render blocking. ST-1, performance audit findings. Lighthouse identified Font Awesome as the primary render-blocking resource (~2200ms FCP savings). The icons used in the app are a small subset — tree-shaking via Lucide would reduce this to near-zero CSS overhead.
