2. Navigation Pain Points

How does a user move between calculators? The site uses a hub-and-spoke model (homepage → individual calculator), but several gaps make inter-calculator navigation difficult.

N-1 · Coverage Calculator Missing from Main Nav

The header navigation shows: Home, Projects → Tiling, Calculators → Unit Converter. The Coverage Calculator is only reachable from the homepage card grid or the sidebar — users who navigate away from the homepage have no top-level route back to it.

Impact: Users cannot discover or return to the Coverage Calculator from the top nav.

N-2 · Wizard Offers No Random-Access to Steps

The 10-step Tiling Wizard only allows Next/Back navigation. The step indicator shows plain text “Step 3 of 10” but is not clickable. There is no visual progress bar.

Tidwell pattern: Sequence Map (2020, ch. 3) recommends clickable progress indicators so users can review and edit earlier inputs without stepping backwards linearly.

Impact: A user on Step 8 who wants to change their tile size on Step 1 must click Back 7 times.

N-3 · Disabled Nav Items with No Explanation

The sidebar shows “Masonry Calculator” and “Concrete & Groundworks” as disabled/greyed links with no tooltip, label, or “Coming soon” badge. They look broken rather than intentional. Meanwhile, homepage cards do show “Coming soon” labels — an inconsistency.

N-4 · Breadcrumbs Not Semantic

Breadcrumbs in SelcoLayout.astro use <div> + <a> + <span> instead of the correct <nav aria-label="Breadcrumb"><ol> structure. Fails WCAG 2.1 and Tidwell’s Breadcrumbs pattern (2020, ch. 3).

N-5 · No “Back to Calculators” Escape Hatch

Once inside the Tiling Wizard, the only way back to the calculator index is the browser back button or the sidebar. No explicit ← Back to calculators link exists at the top of calculator pages.

Tidwell pattern: Escape Hatch (2020, ch. 3) — always provide a clear way out of a flow.

N-6 · Mobile Nav Drawer Has No Focus Trap

MobileMenu.tsx opens a slide-in panel, but keyboard users can Tab past it into the page behind. No Escape key handler to close. Fails Tidwell’s Modal Panel pattern (2020, ch. 3) and WCAG 2.1 focus management.

Three Competing Design Token Systems

The most significant visual design issue is that three separate colour/token systems coexist, one of which is entirely phantom (undefined):

Impact of Phantom Tokens

12 undefined CSS custom properties cause cascading visual failures:

  • Invisible Calculate button (bg-[--color-brand-yellow] → transparent)
  • Invisible focus rings (focus:ring-[--color-brand-blue]/30 → transparent)
  • Invisible error alerts (bg-[--color-destructive]/10 → transparent)
  • No border radius on inputs/cards (rounded-[--radius-input] → 0)
  • Invisible primary results text (text-[--color-brand-blue] → transparent)

These two systems produce subtly different spacing rhythms when SelcoLayout chrome sits alongside Tailwind-powered calculator content.

B-2: Invisible Next Button (Root Cause Confirmed)

Critical · Invisible Buttons

TilingProjectWizard.tsx line 1012 uses bg-brand-blue text-white. Tailwind 4 maps bg-brand-blue → var(--color-brand-blue) → undefined → rgba(0,0,0,0). White text on transparent background = invisible button.

The same issue affects the Calculate button in CalculatorLayout.tsx which uses bg-[--color-brand-yellow].

4. Form Input Review

F-1 · ProjectListsCalculator Has No Labels

ProjectListsCalculator.tsx lines 74–94: all 3 inputs (item name, quantity, unit price) use placeholder text only — no <label> elements. Screen reader users cannot identify these fields.

All other calculators use FormInput with proper <label htmlFor> associations — this component is the sole outlier.

F-2 · Error States Inconsistent
ComponentError HandlingQuality
CalculatorLayout.tsxErrorAlert component (but lacks role="alert")Partial
GroutCalculator.tsxInline validateInputs() with setError()Good
TilingProjectWizard.tsxwindow.alert() for “Please enter valid dimensions”Bad
CoverageCalculator.tsxNo validation at allNone
UnitConverter.tsxNo validation at allNone
F-3 · Input Types Appropriate

All numeric fields use type="number" with min, max, step attributes. Placeholder format is consistent (“e.g. 12.5” pattern). This is well done.

F-4 · No Inline Validation

Errors only appear on Calculate button click, not on field blur. Tidwell’s Same-Page Error Messages pattern (2020, ch. 10) recommends inline feedback next to the problem field, not a banner at the top of the form.

F-5 · No Field-Level Error Association
  • No aria-describedby linking error messages to their input fields
  • No aria-invalid attribute set on fields in error state
  • Screen reader users cannot determine which field caused an error
F-6 · Focus Ring Invisible in CalculatorLayout

FormInput uses focus:ring-2 focus:ring-[--color-brand-blue]/30 — resolves to transparent because --color-brand-blue is undefined. The global .focus-ring utility (ring-2 ring-primary/50) works correctly but CalculatorLayout doesn’t use it.

F-7 · Wizard Steps Lack Validation

Users can click Next on empty steps in the Tiling Wizard without entering any data. No required-field enforcement between wizard steps.

5. Accessibility Basics

A-2 · No Skip Link in SelcoLayout

BaseLayout.astro has a “Skip to content” link, but SelcoLayout.astro — which all calculator pages use — has none. Keyboard users must tab through the entire header, nav, and sidebar before reaching page content.

A-3 · Heading Hierarchy Violations
  • SelcoLayout renders <h1> via .page-title, then calculator components render their own <h1> → duplicate h1 on page
  • TilingProjectWizard.tsx jumps from no <h2> directly to <h3> (lines 972, 983) — skipped heading level
  • CalculatorLayout renders its own <h1> (line 209) — conflicts with SelcoLayout’s
A-4 · ARIA Missing on Interactive Widgets
  • Preset selection buttons (CoverageCalculator) lack aria-pressed
  • Unit toggle buttons (TilingProjectWizard) lack aria-pressed or role="radiogroup"
  • ErrorAlert component lacks role="alert"
  • Step indicator has no aria-live region — step changes not announced
A-5 · Mobile Menu Not Accessible
  • No focus trap — Tab key escapes the menu into the page behind
  • No Escape key listener to close the drawer
  • No aria-modal="true" on the panel
A-6 · Good Patterns Already in Use
  • Form <fieldset>/<legend> groupings provide good screen reader context
  • role="radiogroup" used correctly in AdhesiveCalculator and TilingWizard
  • role="tablist" with aria-selected used correctly in ConversionsCalculator
A-7 · Touch Target Size

Most buttons meet the 44×44px minimum. However, toggle/preset buttons at px-4 py-2 produce ≈32px height — below iOS Human Interface Guidelines minimum.
